# connectors/_core - shared integration-layer docs

This directory carries the shared specifications that every connector under
`reference/connectors/` obeys. It defines how raw data becomes normalized data,
how normalized data becomes derived data, how lineage is tracked, how
exceptions are classified, and how every one of the twenty-seven canonical
workflows activates when data arrives.

## Scope, in one sentence

Canonical contracts, schemas, and prose patterns that the pms, gl, crm, ap,
market_data, construction, hr_payroll, and manual_uploads connectors all
inherit. No vendor code. No dollar figures in prose. snake_case throughout.

## This is not the subsystem _core

Two different `_core/` directories exist in this subsystem and they cover
different scope. Do not conflate.

| Directory | Scope |
|---|---|
| `src/skills/residential_multifamily/_core/` | Canonical subsystem base. Ontology, metrics, routing, approval floor, taxonomy, alias registry, guardrails, schemas. Reads by every pack. |
| `src/skills/residential_multifamily/reference/connectors/_core/` (this directory) | Shared integration-layer contracts. Layer design, lineage model, normalization patterns, field-mapping template, workflow activation map, exception taxonomy, lineage and override log schemas. Reads by every connector. |

If you are working on a metric contract, an object definition, a routing rule,
or a skill pack manifest, you are in the subsystem `_core/`. If you are working
on ingestion, normalization, lineage, or how workflows light up when inputs
arrive, you are here.

Both directories are immutable to the tailoring skill (see subsystem
`_core/BOUNDARIES.md`). Both are source-of-truth for their scope.

## Connectors in scope

The subsystem defines eight connectors. Six are built today. Two are under
parallel construction.

| Connector | Status | Notes |
|---|---|---|
| `pms/` | Built | Rent roll, leases, lease events, charges, payments, delinquency, leads, tours, applications, work orders, turns. |
| `gl/` | Built | Chart of accounts, account mapping, actuals, budget, forecast, variance, capex actuals, commitments. |
| `crm/` | Built | Prospect and resident CRM: lead interactions, campaign sources, resident communications, service requests. |
| `ap/` | Built | Vendor master, invoices, contracts, commitments, purchase orders, payment status. |
| `market_data/` | Built | Rent comps, concession observations, occupancy, payroll, labor, materials, utility references. |
| `construction/` | Built | Development budgets, estimate line items, bid packages, commitments, change orders, draw requests, schedule milestones, punch items. |
| `hr_payroll/` | Parallel build | Staffing plans, approved positions, site roster, headcount, payroll actuals, vacancy events, turnover events. |
| `manual_uploads/` | Parallel build | Operator- and manager-submitted files that arrive outside direct system integration. File-drop templates, attestation metadata, template versioning. |

Every document in this directory references all eight. Files and patterns that
only apply to one connector live under that connector's own directory, not here.

## How skill packs read from here

Skill packs do not read these documents at runtime. Packs read:

- `_core/` canon for metric contracts and objects.
- `reference/normalized/` and `reference/derived/` for data.
- Their own `reference_manifest.yaml` for declared reads.

This directory is source-of-truth for the humans and agents that build and
maintain the connector layer, and for the tests that enforce contracts
(`tests/test_connector_contracts.py`). When the subsystem acquires a new
connector, it is built to match the patterns here before any skill pack can
list it as a read source.
