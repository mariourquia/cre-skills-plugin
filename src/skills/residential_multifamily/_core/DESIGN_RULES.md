# Design Rules (non-negotiable)

Any skill pack, overlay, template, or reference file that breaks one of these rules must be rejected. Tests in `tests/residential_multifamily/` enforce as many of these as can be checked mechanically; the rest are enforced in review.

## Rule 1 — Externalized reference layer

Every mutable figure lives in `reference/`, not in skill prose.

**Mutable figures include but are not limited to:**

- Market rents, concessions, occupancy benchmarks, delinquency norms
- Renewal conversion benchmarks, tour conversion benchmarks
- Staffing ratios, unit turn benchmarks, make-ready durations
- Material pricing (lumber, flooring, paint, appliances, fixtures)
- Labor rates, vendor rate cards, utility benchmarks
- Insurance, tax, payroll assumptions
- Capex line-item libraries, unit turn cost libraries
- Development budget norms, construction duration norms
- Contingency assumptions, cost escalation assumptions
- Rent comp data, unit mix benchmarks
- Financial thresholds that vary by market, segment, or stage

**A skill prose file must not contain:**

- A specific dollar figure other than a format example clearly tagged as illustrative.
- A specific percentage benchmark other than a format example clearly tagged as illustrative.
- An implied fixed ratio (e.g., "1 leasing agent per 100 units"). Ratios live in reference.

Tests enforce this by scanning role/workflow/overlay `.md` files for numeric density patterns outside fenced code blocks.

## Rule 2 — Canonical core, overlays on top

Canonical concepts are defined once in `_core/`. Skills and overlays reference canonical definitions; they do not redefine them.

**Allowed:**

- An overlay may *constrain* a canonical concept (e.g., middle-market overlay says `renewal_offer_rate` target range is tighter than the general default).
- An overlay may add new segment-specific concepts (e.g., affordable overlay adds `rent_limit_compliance_rate`).
- An overlay may override a threshold or default; the override must cite the canonical concept and the reason.

**Forbidden:**

- Silently redefining a canonical metric or object.
- Creating a parallel metric with the same meaning under a different name.
- Presenting a metric in a pack without citing its canonical definition.

Tests enforce this by verifying every metric name in a skill's `metrics.md` appears in `_core/metrics.md` or is explicitly declared as an overlay-added metric in the overlay's manifest.

## Rule 3 — Human-readable, diff-friendly, agent-editable formats

- Narrative definitions, workflow logic, playbooks, templates: Markdown.
- Schemas, manifests, routing rules, configuration: YAML.
- Tabular comps, benchmark packs, pricing histories, line-item estimate libraries, sample operating data: CSV.

JSON is acceptable for machine-generated artifacts (test output, manifests generated by tools). Do not author JSON by hand if YAML will do.

No binary formats. No proprietary formats. No formats that require a tool to diff.

## Rule 4 — Every metric is a data contract

See `schemas/metric_contract.yaml` for the enforced schema. Every canonical metric in `_core/metrics.md` must declare:

- `name`, `slug`, `description`, `grain`, `time_basis`, `scenario`, `unit`,
- `numerator`, `denominator` (if a rate or ratio),
- `filters_default`, `inclusions`, `exclusions`, `rollup_rule`, `null_handling`,
- `source_fields`, `qa_rule`, `reconciliation_rule`,
- `open_questions` (may be empty).

A metric is incomplete until all required fields are present. Tests fail if any required field is missing.

## Rule 5 — No duplicate meaning

Every metric has exactly one canonical definition. Aliases are permitted and are tracked in `alias_registry.yaml`; silent parallel definitions are prohibited.

Examples of duplicate-meaning hazards that `alias_registry.yaml` must disambiguate:

- `occupancy`: physical vs leased vs economic vs preleased vs notice-adjusted.
- `rent_growth`: same-store vs portfolio-wide vs market vs in-place vs new-lease vs renewal vs blended.
- `make_ready_days`: notice-to-ready vs vacant-to-ready vs vacant-to-leased.
- `delinquency`: gross vs net vs 30+ vs 60+ vs total receivable vs current-month.
- `turnover`: unit turns vs resident turnover vs annualized vs rolling-12.

Tests enforce by scanning metric slugs for collisions and verifying aliases are registered.

## Rule 6 — Progressive disclosure

A request is resolved through the routing hierarchy before packs load. Skills and templates are not eagerly loaded; they are surfaced by the router.

Hierarchy:

1. Asset class: `residential_multifamily` (this subsystem).
2. Segment (market-positioning, conventional only): `middle_market`, `luxury`.
3. Regulatory program: `none` (default), `lihtc`, `hud_section_8`, `hud_202_811`, `usda_rd`, `state_program`, `mixed_income`. Composes additively with segment.
4. Form factor: `garden`, `walk_up`, `wrap`, `suburban_mid_rise`, `urban_mid_rise`, `high_rise`.
5. Lifecycle stage: `development`, `construction`, `lease_up`, `stabilized`, `renovation`, `recap_support`.
6. Management mode: `self_managed`, `third_party_managed`, `owner_oversight`.
7. Role: (see `taxonomy.md`).
8. Task or workflow: (see `workflows/`).
9. Geography: market, submarket (via `reference/normalized/markets/`).
10. Output type: `memo`, `kpi_review`, `checklist`, `estimate`, `operating_review`, `scorecard`, `dashboard`, `email_draft`.
11. Decision severity: `informational`, `recommendation`, `action_routable`, `action_requires_approval`.

Routing rules live in `routing/`. A request that cannot resolve a required axis must ask the user or fail gracefully with a clear prompt for the missing axis.

Boundary rules that govern what content may live in each layer (core, segment, regulatory, market, org) are defined in `_core/BOUNDARIES.md` and enforced by `tests/test_boundary_rules.py`.

## Rule 7 — Auditability and update history

Every change to a reference value must support:

- `old_value`, `new_value`
- `source_name`, `source_type`, `source_date`
- `as_of_date`, `effective_start`, `effective_end` (if applicable)
- `proposed_by` (agent ID or human), `approved_by`
- `confidence` (one of `low | medium | high | verified`)
- `reason_for_change`
- `affected_skills` (optional, automatable)

See `schemas/change_log_entry.yaml` for the enforced schema and `change_log_conventions.md` for examples.

## Rule 8 — Controlled autonomy

Default autonomy: the system may draft, analyze, route, summarize, monitor, and recommend.

Explicit approval gates (see `approval_matrix.md`) are required for:

- Legal notices and eviction-sensitive actions.
- Fair housing– or screening-sensitive actions.
- Resident disputes with legal exposure.
- Safety-critical maintenance decisions.
- Licensed engineering or code compliance judgments.
- Financial disbursements above configurable thresholds.
- Contract awards and material procurement above thresholds.
- Lease deviations, concessions beyond policy, non-standard payment plans.
- Lender-facing or investor-facing submissions marked `final`.
- Any action explicitly marked `human_approval_required` by policy.

The system must refuse to execute a gated action without a linked `approval_request` record in `approved` status. Tests verify that gated-action paths all check for an approval record.

## Rule 9 — Sample data is tagged

Every example reference file must carry `status: sample | starter | illustrative | placeholder` at the record or file level. Skills must not cite sample data as fact when producing outputs; they must surface the `status` tag.

## Rule 10 — No silent failure

Missing data fails loudly. A skill that cannot find a required reference must:

1. Name the missing reference by category and scope (e.g., `reference/normalized/market_rents.csv for market=Charlotte, segment=middle_market`).
2. State what the default behavior is (refuse, use prior period, use portfolio average, ask human).
3. Log the gap for the tailoring skill's missing-docs queue.

Silent defaulting to zero, to the portfolio average, or to "the last thing we saw" without surfacing is prohibited.
