# v4.3.0 — residential_multifamily graduation + v4.3 hardening close

## Summary

Three tracks land in this release:

1. **residential_multifamily graduates from `beta_rc` to `stable_pending_shakedown`.** The subsystem moves from `status: beta_rc` (v0.6.0) to `status: stable_pending_shakedown` (v1.0.0-rc1). Tailoring Pass 2 Objective 4 closes: the four items previously marked **Not implemented** in `docs/tailoring_capability_matrix.md` (approval-floor refusal, canonical-definition redefinition refusal, preview-bundle YAML emission, missing-doc blocker) are all now **Implemented** and under test on `main`. The subsystem is code complete and refusal-on-missing-input contracts are active; only the first operator shakedown log remains before graduation to `status: stable`.
2. **Portable ZIP structural smoke coverage.** `tests/install_smoke/test_portable_zip.py` and `.github/workflows/portable-zip-smoke.yml` validate the packaged `dist/cre-skills-portable.zip`: ZIP opens cleanly, skills tree mirrors `src/skills/` minus the portable-excluded `residential_multifamily` subsystem, each extracted `SKILL.md` carries the portable frontmatter contract, and no MCP server / orchestrator / Python calculator leaks in. Cross-runtime invocation on Codex / Gemini / Grok / Manus remains an explicit gap.
3. **Canonical Desktop marketplace caveat + preview-mode gate.** The "Chat tab's 'Add marketplace' is not supported" caveat is pinned verbatim across five install surfaces with a byte-level parity assertion in `tests/test_release_version_parity.py`. A repo-wide `tests/test_preview_mode_gate.py` walks every `src/skills/**/SKILL.md` and enforces the Release maturity section + banner text per preview sub-status; the subsystem-scoped counterpart is extended to admit and pin the `stable_pending_shakedown` banner.

This is not a new-feature release. It is a status-reconciliation and release-truthfulness step: internal code was already in place; docs, statuses, and versions across public surfaces now match.

The `residential_multifamily` subsystem is **stable_pending_shakedown** — its refusal-on-missing-input contracts stay active and its output carries a `Stable, awaiting shakedown` banner until the first operator shakedown log lands. The portable ZIP is **experimental (structural CI)**; cross-runtime invocation is still unsupported. Claude Desktop Chat tab's "Add marketplace" dialog remains explicitly not supported; the Claude Code CLI marketplace path is.

## What changed

### residential_multifamily graduation

- `src/skills/residential_multifamily/SKILL.md`: frontmatter `status: beta_rc` → `status: stable_pending_shakedown`; `version: 0.6.0` → `version: 1.0.0-rc1`. Release maturity section updated to carry the `Stable, awaiting shakedown` banner (per `docs/PREVIEW_MODE.md`), and explicitly disclaims the second-refusal layer that `beta_rc` would have required (shakedown status relies on the existing contract machinery: sealed-close floor, finance-critical placeholder scanner, executive output contract, tailoring guards).
- No change to underlying subsystem behavior — the graduation is a status reconciliation, not a new code path. Refusal-on-missing-input contracts remain load-bearing.

### Tailoring Pass 2 Obj 4 close

The four items previously listed as **Not implemented** in `docs/tailoring_capability_matrix.md` are all live:

- `tailoring_tui.py::guard_approval_floor_not_lowered()` — compares every numeric approval-threshold path in the proposed overlay against the current floor; refuses on any drop. `ApprovalFloorGuardTests` (3 tests).
- `tailoring_tui.py::guard_no_canonical_redefinition()` — parses `_core/ontology.md` and `_core/metrics.md` headers, refuses any proposed-overlay key at canonical scope that carries a `redefinition` or `rename` directive. `CanonicalRedefinitionGuardTests` (2 tests).
- `tailoring_tui.py::emit_preview_bundle()` — writes `tailoring/sessions/<org_id>/<session_id>__preview.yaml` with session / diff_summary / diff_entries / proposed_overlay / guard_violations sections. `PreviewBundleTests`.
- `tailoring_tui.py::guard_missing_doc_blockers()` — walks every question-bank trigger, refuses if any references a `doc_catalog.yaml` slug with no entry. Tolerates both string and dict trigger shapes. `MissingDocBlockerGuardTests` (2 tests).

`docs/tailoring_capability_matrix.md` "Deliberate next steps" block rewritten to reflect Obj 4 closure; only legacy-bank retirement and role-based bank filtering remain as open follow-ups (tracked outside Obj 4).

### Preview-mode gate — repo-wide

`docs/PREVIEW_MODE.md` defines a fourth status, `stable_pending_shakedown`:

| Status | Banner | Final-marked refusal layer |
|---|---|---|
| `experimental` | PREVIEW / STAGING stamp | Yes — refuses without human ack. |
| `beta_rc` | PREVIEW / STAGING stamp | Yes — refuses without human ack. |
| `stable_pending_shakedown` | `Stable, awaiting shakedown` | **No** — refusal contracts already active; preview gate does not layer a second refusal. |
| `stable` | (no banner) | Not applicable. |

- `tests/test_preview_mode_gate.py` (repo-wide, new in v4.3): walks every `src/skills/**/SKILL.md`. Asserts every preview-status skill carries the Release maturity section + required markers; asserts banner text per sub-status; asserts shakedown skills do not re-import PREVIEW / STAGING stamp language (which would silently layer a second refusal).
- `src/skills/residential_multifamily/tests/test_preview_mode_gate.py` (subsystem, extended in v4.3): admits `stable_pending_shakedown`, pins the shakedown banner tokens, and explicitly asserts no shakedown skill carries the beta_rc "not eligible for final-marked use without human acknowledgement" phrase.

### Portable ZIP structural smoke

- `tests/install_smoke/test_portable_zip.py`: validates the packaged `dist/cre-skills-portable.zip` directly.
  - ZIP opens and passes CRC.
  - Every non-excluded source skill (`src/skills/<slug>/SKILL.md` minus `residential_multifamily`) has a matching `skills/<slug>/SKILL.md` in the ZIP.
  - Each extracted SKILL.md parses and carries `name` + `description` (the portable contract). `slug`, `status`, `version`, `category`, `targets` are stripped by design per `config/targets/portable.yaml`; the test pins their absence as a guard against normalizer regressions.
  - No `mcp-server.mjs` at the top level.
  - No `orchestrators/` tree.
  - No `.py` files anywhere in the ZIP.
- `.github/workflows/portable-zip-smoke.yml`: triggered on `pull_request` for paths in `tools/package/`, `tools/build*`, `tools/normalize/`, `config/targets/portable.yaml`, `src/skills/**`, and the smoke test + workflow files themselves; plus `workflow_dispatch`. Runs `npx tsx tools/build.ts --target portable`, `npx tsx tools/package/package-portable.ts`, then `pytest tests/install_smoke/test_portable_zip.py`.
- `TestPortableZipCrossRuntimeInvocation` is `@unittest.expectedFailure` with a message pointing at `docs/install_smoke_test_matrix.md` — structural != cross-runtime; a real cross-CLI harness is a separate gap tracked in the matrix.
