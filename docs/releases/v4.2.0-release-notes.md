## Summary

Two tracks land in this release:

1. **residential_multifamily hardening pass 2 close.** Moves the subsystem from `status: draft` (v0.5.0) to `status: beta_rc` (v0.6.0). Closes the three objectives deferred from pass 1: sealed-close gating (Obj 5), a finance-critical placeholder scanner (Obj 6), and an executive output contract (Obj 8).
2. **Release truthfulness + packaging coherence.** Canonicalizes the Claude Code CLI marketplace install path, corrects release-asset filenames in every install doc, removes the last stale `v4.0.0` / `112 skills` strings from user-visible installer banners, makes the MCP server read its version from `plugin.json` instead of carrying a hardcoded string, and adds a docs/surface/version parity test so this class of drift fails CI on the next release.

This is not a feature release. It is a hardening step: fewer silent failure modes, more declarative gates, auditable output contracts, and more honest public surfaces.

The `residential_multifamily` subsystem remains **beta RC** and its final-marked workflows continue to refuse when required inputs are absent. The portable ZIP for Codex / Gemini / Grok / Manus remains **experimental** and is not in the CI smoke matrix. Claude Desktop Chat tab's "Add marketplace" dialog is explicitly not supported; the Claude Code CLI marketplace path is.

## What changed

### Sealed-close gating (Obj 5)

Period-grade workflows (monthly operating review, reforecast, budget build, quarterly LP letter, executive operating summary) will now refuse to run against draft GL data.

- `_core/schemas/period_seal.yaml` canonicalizes the ordering `draft < soft_close < hard_close < locked` plus `as_of`, `close_lock_timestamp`, `budget_version`, `reforecast_version`.
- `_core/final_marked_workflows.yaml#period_grade_workflows` enumerates 6 slugs with their minimum close-status floor:
  - `hard_close + close_lock_timestamp`: `executive_operating_summary_generation`, `quarterly_portfolio_review`
  - `soft_close`: `monthly_property_operating_review`, `monthly_asset_management_review`, `reforecast` (with `budget_version` required), `budget_build`
- Six workflow `reference_manifest.yaml` files declare `required_period_seal` aligned with the registry.
- `_core/schemas/reference_manifest.yaml` extended to admit the new field.
- 5 new tests in `tests/test_period_seal_gating.py` enforce the contract: registry non-empty, manifest conformance, stray-declaration detection, hard-close implies `close_lock_timestamp`.

### Finance-critical placeholder scanner (Obj 6)

Hidden placeholder rows (`TBD`, `TODO`, `FIXME`, `XXX`, `PLACEHOLDER`, `TKTK`) in a reference CSV read by a final-marked workflow are the main silent-failure hazard for board/LP/lender output. This release closes the hole with an explicit-labeling rule.

- `_core/reference_data_integrity.md` documents the rule: any reference row containing a placeholder token must also carry an explicit label (`status=placeholder|tbd|todo|deferred`, `confidence=placeholder|low_placeholder`, `source_type=placeholder`, `placeholder=true`, or `placeholder_row=true`).
- 4 new tests in `tests/test_finance_placeholder_scanner.py` run the scanner on every CSV read by a final-marked workflow manifest and verify the synthetic positive/negative cases.

### Executive output contract (Obj 8)

Final-marked output MUST lead with the verdict and tag every numeric cell with its source class. The contract is a doc + cross-references + a worked example + a test.

- `_core/executive_output_contract.md` defines three rules: verdict-first block (recommendation / 3-bullet rationale / confidence / materiality / next action), source-class labels on every numeric cell (`[operator]` / `[derived]` / `[benchmark]` / `[overlay]` / `[placeholder]`), and refusal-artifact shape when a required input is absent.
- Four final-marked workflow SKILL.md files (`executive_operating_summary_generation`, `investment_committee_prep`, `quarterly_portfolio_review`, `executive_pipeline_summary`) reference the contract explicitly.
- The canonical example (`executive_operating_summary_generation/examples/ex01_*.md`) demonstrates all five source classes plus the verdict block.
- 4 new tests in `tests/test_executive_output_contract.py` enforce: contract doc exists with all three rule markers, every final-marked SKILL references the contract, at least one example demonstrates the full source-class set, canonical example carries the verdict markers.

### Release truthfulness + packaging coherence

- Canonicalizes the Claude Code CLI marketplace install flow across README, `docs/INSTALL.md`, `docs/install-guide.md`, `docs/install-desktop.md`, `docs/install-cowork.md`, `docs/WHAT-TO-USE-WHEN.md`, `docs/release-checklist.md`, and the install section the release workflow emits into the published release body. Alternative install commands are retained only inside explicit advanced/dev subsections. Before this pass, five conflicting install commands appeared across public docs.
- Clarifies that the Claude Code CLI marketplace (`claude plugin marketplace add mariourquia/cre-skills-plugin && claude plugin install cre-skills@cre-skills`) is supported while the Claude Desktop Chat tab's "Add marketplace" dialog is not. The two are different surfaces.
- Corrects the release-asset filenames users are told to download. The release workflow renames artifacts to `cre-skills-plugin-${TAG_NAME}.{dmg,-setup.exe,tar.gz,zip}` at publish time; docs previously pointed at the pre-rename names (`cre-skills-v${VERSION}.dmg`, `cre-skills-v${VERSION}-setup.exe`), which did not match any published asset.
- Fixes the marketplace install suffix emitted by the release workflow so the installed plugin key resolves against the marketplace name declared in `.claude-plugin/marketplace.json` (`cre-skills`). Prior copy used a trailing `-plugin` suffix that did not resolve.
- Replaces stale "112 skills", "v4.0.0" installer banners, and the hardcoded `plugin_version: "4.0.0"` inside `src/mcp-server.mjs` with the real current values; `mcp-server.mjs` now reads the version from `.claude-plugin/plugin.json` so it cannot drift again.
- Adds `tests/test_release_version_parity.py` to fail CI when versions, install commands, skill counts, or release-asset filenames disagree across manifests, installer scripts, and install docs.
- Flips `docs/releases/v4.2.0-release-notes.md` frontmatter to `status: pending` until the tag actually exists; it is promoted to `status: released` in a follow-up commit once the release workflow completes.

## Migration notes

1. **Manifests** may now declare an optional `required_period_seal` top-level field. Workflows without it continue to work unchanged.
2. **Period-grade workflows** will REFUSE when `close_status` is below the declared floor. Floors are deliberately conservative.
3. **Final-marked output** is now required to carry source-class tags on numeric cells. Existing authors should read `_core/executive_output_contract.md` before their next update.
4. No breaking changes to the router, to tailoring, or to the MCP server.
